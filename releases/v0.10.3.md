# Release v0.10.3

**Release Date:** 2026-02-03

## Summary

This release delivers significant performance improvements (~17% faster on large files) and adds millisecond precision support for timestamp parsing and filtering.

## Performance Improvements

**Benchmark results** (277MB / 1.43M line access log):
- **Baseline (v0.10.2):** ~16.6s average
- **New (v0.10.3):** ~13.7s average
- **Improvement:** 17.5% faster

### DateTime to Time::Piece Migration (#43)

Replaced the DateTime.pm library with Time::Piece and Time::Local for timestamp parsing. DateTime.pm was identified as the single largest performance bottleneck, accounting for ~18% of runtime due to heavy dependencies (Specio type checking, Eval::Closure, Params::ValidationCompiler).

**Changes:**
- Use `Time::Local::timegm()` for epoch conversion instead of DateTime objects
- Removed DateTime dependency from cpanfile (smaller installation footprint)
- Timestamp cache now stores numeric epoch values instead of objects

### Processing Loop Optimizations (#48)

Removed unnecessary overhead in the main log processing loop:

- Removed `visualize_carets()` function (1.54s / 3.5% savings)
- Removed temporary customer-specific substitutions
- Replaced Math::Round with built-in `int($x + 0.5)`
- Conditionalized metric extraction substitutions

## New Features

### Millisecond Precision Support (#43)

The `-ms` flag now provides true millisecond precision for timestamp parsing and filtering:

- **Timestamp parsing:** Extracts and preserves milliseconds from log timestamps (supports both `.` and `,` separators, 1-6 fractional digits)
- **Time buckets:** Uses integer milliseconds internally to avoid floating-point precision issues
- **Time filters:** `-st` and `-et` options now accept millisecond precision

**Usage examples:**
```bash
# View log timeline with 100ms time buckets
./ltl -ms -bs 100 logs/application.log

# Filter to a specific sub-second time range
./ltl -ms -bs 100 -st "08:15:30.500" -et "08:15:31.250" logs/application.log

# Full date with milliseconds
./ltl -ms -st "2025-04-10 12:34:56.789" logs/application.log
```

## Bug Fixes

- Fixed bucket alignment issue where sub-second buckets displayed imprecise boundaries (e.g., `.099` instead of `.100`) due to floating-point arithmetic
