# Release v0.14.4

## What's New

- Redesign final consolidation pass with sliding window architecture — replaces unbounded streaming simulation with two-pass bounded design operating directly on `%log_messages`. Pass 1: sorted iteration with 1000-key sliding window (S3 + S4). Pass 2: conditional cleanup sweep against newly created patterns. No data duplication, bounded trigram lifecycle per window (#150)
- Remove 2M line auto-disable for final pass — the new bounded architecture is safe to run on any file size (#150, #144)
- Rename `--final-ceiling` to `--group-ceiling` (`-gc`) — user-facing control for consolidation scope: messages with more than N occurrences skip pairwise discovery but still match existing patterns (#150)
- Remove internal consolidation tuning options from help and documentation — `--consolidation-trigger`, `--consolidation-ceiling`, `--consolidation-max-patterns`, `--final-threshold`, `--no-final-pass` remain functional as hidden options (#150)
- Track transient consolidation structures (trigram index, posting lists, per-key trigrams) in `-mem` HWM output — previously invisible, these account for up to 87% of peak RSS during consolidation (#171)
- Add `MEMORY_FINAL` end-of-execution memory snapshot and hash entry counts to `-V -mem` output (#150, #166)
- Split S4 verbose metric into "S4 Pairwise discovery" and "S4 Re-scan absorbed" for consolidation observability; add cross-cluster merge count; add detailed process flow diagram to feature documentation (#162)
- Benchmark script now passes `--terminal-width 200` and `-bs 60` (when not explicitly set) for consistent results across terminal environments. Added `--options "<ltl options>"` passthrough for injecting additional options into all benchmark runs (#167)

## Bug Fixes

- Fix `--terminal-width` not applied before log key truncation — CLI overrides were parsed after auto-detected defaults (#157)
- Fix log key truncated to terminal width breaking consolidation on long messages — when `-g` is active, use 350-char cap (same as CSV) instead of terminal width, preserving UUIDs and other variable content needed for similarity detection (#158)
- Fix cross-cluster pattern merge testing subsumption in wrong direction — more general patterns now correctly absorb more specific ones (#164)
- Adjust CV decimal places based on magnitude — values under 10 show 2 decimals, 10-99 show 1 decimal, 100+ show no decimals (#152)

### Known Regressions

**Non-consolidation memory (all tests):** RSS increased 4-77% across non-consolidation tests. This is caused by the v0.14.3 baseline being captured at terminal width 120 (non-TTY) while v0.14.4 was captured at width 200. Wider terminal produces longer log keys and more unique entries in `%log_messages`. Not a code regression — benchmark script now enforces consistent `--terminal-width 200` for future comparisons (#166, #167).

**Small-file consolidation (humungous-log, 288K lines):** Timing increased from 3.7s to 10.4s (+177%), memory from 127 MB to 246 MB (+94%). This is the cost of the #158 correctness fix — log keys are now 350 chars when `-g` is active (previously truncated to terminal width 120), preserving UUIDs and variable content needed for consolidation. On files with 350-char messages and 286K unique keys, longer keys produce ~3× more trigram data during checkpoint processing. The consolidation itself works correctly — S1 absorbs 98.2% of keys. An adaptive message length cap (#174) would reduce this overhead on files with shorter messages.
